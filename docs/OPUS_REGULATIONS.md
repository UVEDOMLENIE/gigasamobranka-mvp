# Регламент Опуса — «Выход на свет»

> Метафора: Опус — это Зверь из фильма «Сплит». Появляется редко, в особых условиях, оставляет
> чёткий след, уходит обратно в тень. Между его появлениями работают другие модели (Kimi, SWE-1.6).
> Пользователь — режиссёр, решает когда вызвать кого.

---

## Когда вызывать Опуса

✅ **Вызывать**:
- Спринт закончен → нужен отчёт и новые задачи.
- Дешёвая модель застряла на одной баге 30+ минут.
- Нужно архитектурное решение (миграция, новая фича, big refactor).
- Бага непонятная — нужна диагностика по логам.
- Перед сдачей — финальный аудит.

❌ **НЕ вызывать**:
- Изолированная правка одного файла (Kimi справится).
- Линт / форматирование / переименование переменных.
- Добавить кнопку / изменить текст / переставить блок.
- "Прогнать smoke и закоммитить" — это сама модель должна.

---

## Что Опус делает за один выход

**Обязательная программа:**

1. Читает в этом порядке:
   - `docs/BACKLOG.md` — новые идеи/баги от юзера
   - `docs/sprints/NN-name/REPORT.md` — последний отчёт исполнителя
   - `docs/CURRENT_STATE.md` — что работает сейчас
   - `git log --oneline -20` — что коммитилось

2. Принимает решения:
   - Какие задачи из backlog уйдут в новый спринт.
   - Какие отложатся / отменятся.
   - Какие выявленные баги критичны.

3. Создаёт артефакты:
   - `docs/sprints/NN+1-name/TASKS.md` — новый спринт.
   - Обновляет `docs/BACKLOG.md` — отмечает что переехало `[→ Sxx]`.
   - Обновляет `docs/ROADMAP.md` если статус изменился.
   - Обновляет `docs/CURRENT_STATE.md` если фичи добавились / убрались.

4. Коммитит + пушит одним коммитом:
   ```
   git commit -m "opus: завершён S0N, открыт S0N+1 — <тема спринта>"
   ```

5. Уходит в тень.

---

## Формат файлов

### `TASKS.md` спринта

- **≤ 200 строк**. Если больше — слишком много задач, разделить на 2 спринта.
- **3–6 задач** в спринте. Каждая = 30–60 мин работы Kimi.
- **Формат каждой задачи**:
  ```
  ### T1 — <короткое название> (P0/P1/P2)

  **Проблема**: 1–2 предложения.
  **Файл**: путь.
  **Что делать**: 3–5 шагов.
  **Acceptance**: 2–4 пункта.
  **Коммит**: точная строка для git commit -m
  ```
- В конце — таблица «Очередь» с приоритетами.
- В шапке — короткие правила (≤ 6 пунктов).

### `REPORT.md` спринта

- **≤ 100 строк**.
- Секции (обязательно):
  - **Сделано** — таблица: коммит, что сделано.
  - **Не сделано** — что и почему.
  - **Проблемы** — что вылезло в процессе.
  - **Следующее** — что надо Опусу решить (если есть).

### `BACKLOG.md`

- Плоский список с тегами `[ ]`, `[→ Sxx]`, `[x]`, `[~]`.
- Без вложенности — чтобы юзер мог быстро вкинуть мысль.
- Опус разгребает при каждом выходе.

### `ROADMAP.md`

- Таблица спринтов: # / название / статус / цель.
- Архитектура (для презентации).
- Стек технологий.
- Риски.

## Структура `docs/`

```
docs/
├── BACKLOG.md                 ← юзер кидает мысли сюда
├── ROADMAP.md                 ← общий план до сдачи
├── OPUS_REGULATIONS.md        ← этот файл
├── CURRENT_STATE.md           ← что работает сейчас
├── KNOWN_ISSUES.md            ← известные косяки
├── ORG_REQUIREMENTS.md        ← требования организаторов
├── DEMO_SCRIPT.md             ← сценарий показа жюри
├── CLOUD_HANDOFF.md           ← инструкция для Devin Cloud
└── sprints/
    ├── 01-foundation/
    │   ├── TASKS.md           ← задачи спринта
    │   └── REPORT.md          ← отчёт после спринта
    ├── 02-stability/
    │   ├── TASKS.md
    │   └── REPORT.md          ← добавится по завершении
    └── ...
```

**Имена папок спринтов**: `NN-короткая-тема` (snake-case или kebab-case).
Примеры: `01-foundation`, `02-stability`, `03-polish-submission`.

---

## Правила для Опуса самого

1. **Не делай работу за Kimi.** Не пиши код в TASKS.md больше 20 строк на задачу — это уже не ТЗ, а решение.
2. **Не бюрократствуй.** Лучше короткое и понятное, чем длинное и формальное.
3. **Не дублируй файлы.** Если такая инфа есть в `CURRENT_STATE.md` — ссылайся, не копируй.
4. **Не жги токены.** Один выход = один батч изменений → один коммит → выход.
5. **Проверь smoke перед уходом**: `npm run smoke` должен быть green.
6. **НЕ коммить ключи**, `.env.local`, `local.db`, `node_modules/`.
7. **Никаких реальных имён** в коде/коммитах/документах.

## Шаблон коммита Опуса

```
opus: завершён SNN, открыт SNN+1 — <тема>

- закрыто: какие задачи из SNN успешно
- перенесено: какие в SNN+1
- баклог: что отложено
```

Пример:
```
opus: завершён S01, открыт S02 — Stability

- закрыто: T1-T6 (UI улучшения, /library, DOCX, тайминг)
- перенесено: новое — Scarlex retry, mock fix, deploy
- баклог: 4 идеи переехали в S02, остальные отложены
```

## Триггер вызова Опуса для юзера

Юзер пишет одну из фраз:
- *"Опус, выныривай"*
- *"Опус, нужен спринт-репорт"*
- *"Опус, что дальше"*
- *"Опус, разберись"* (для бага)

Опус выходит, делает программу выше, уходит.
